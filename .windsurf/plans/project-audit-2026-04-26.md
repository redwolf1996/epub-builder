更新时间：2026-04-26

## 1. 高优先级问题

### 1.1 Markdown 预览存在注入风险

状态：❌

位置：
- `src/utils/markdown.ts`
- `src/components/preview/MarkdownPreview.vue`

问题：
- `markdown-it` 开启了 `html: true`
- 渲染结果直接通过 `v-html` 注入预览区
- 当前没有任何 HTML 白名单、属性过滤或 URL 过滤

影响：
- 恶意或误写内容可以直接注入任意 HTML
- 在桌面端这不只是“预览错乱”，而是实际的安全和稳定性风险
- 导出 EPUB 时也会把不受控 HTML 带进去，导致阅读器兼容性下降

计划：
- 将 Markdown 渲染策略拆成“编辑预览”和“导出 HTML”两条受控流水线
- 默认关闭原始 HTML，或至少引入允许标签/属性白名单
- 对链接、图片、内联样式做协议和属性过滤

### 1.2 导出成功反馈时机错误，失败路径可能被吞掉

状态：❌

位置：
- `src/composables/useEpub.ts`
- `src/utils/epub.ts`
- `src/pages/Editor.vue`

问题：
- `downloadEpub()` 在 Tauri 分支里直接调用异步 `downloadEpubTauri()`，但自身不是 `async`
- 上层 `handleExport()` 在文件真正写完前就提示成功并发送通知
- 保存对话框取消、写文件失败、权限异常都可能无法正确反馈到 UI

影响：
- 用户看到“导出成功”，但文件可能未保存
- 错误处理不完整，桌面端体验不可信

计划：
- 将下载流程改为统一 `async`
- 明确区分“生成成功”“选择保存路径”“写盘成功”“用户取消”
- 只有写盘完成后才显示成功消息和系统通知

### 1.3 图书元数据更新后，本地状态不同步

状态：❌

位置：
- `src/composables/useBook.ts`
- `src/stores/book.ts`
- `src/pages/Settings.vue`

问题：
- `updateBookMeta()` 只写 Dexie，不同步更新 `books.value` 和 `activeBook.value`
- 设置页保存后，书架列表、顶部标题、当前活动书籍信息可能保持旧值，直到重新加载

影响：
- UI 状态不一致
- 很容易诱发“明明保存了但没生效”的感知 bug

计划：
- 把图书更新封装成“写库 + 更新内存状态”的单一入口
- `books`、`activeBook`、页面显示数据保持同源

### 1.4 章节树缺少结构约束，存在非法层级风险

状态：❌

位置：
- `src/composables/useChapter.ts`
- `src/components/editor/ChapterNode.vue`

问题：
- 当前仅按 `parentId` + `order` 维护树
- 没有防止“节点拖入自己的后代”这类环形结构
- `moveChapterToParent()` 和排序逻辑没有层级合法性校验

影响：
- 章节树一旦出现环，渲染、导出、删除递归都会失控
- 后续做大纲折叠、批量操作、导出 TOC 时会越来越脆弱

计划：
- 增加树结构约束校验
- 在拖拽落点阶段拦截非法重排
- 数据层增加 `canMoveChapter()` 这类纯函数校验

### 1.5 保存流分散在页面和 store，容易产生竞态

状态：❌

位置：
- `src/stores/editor.ts`
- `src/pages/Editor.vue`
- `src/pages/Settings.vue`
- `src/composables/useChapterManager.ts`

问题：
- 编辑器内容保存、设置页保存、切章时 flush/cancel/save 的编排分散在多处
- 页面自己管理保存状态，store 也管理保存状态
- 存在“先 flush，再 cancel，再切章”的重复和顺序耦合

影响：
- 维护成本高
- 后续加“退出确认”“多窗口”“历史记录”“冲突恢复”会很难收敛

计划：
- 把保存流程集中到 composable/store
- 页面只发出“切章”“离开页面”“立即保存”意图，不直接编排底层顺序
- 保存状态统一为可复用的状态机

## 2. UX / UI 改进点

### 2.1 新建书籍流程过于粗糙

状态：❌

问题：
- 首页新建只输入标题，随后再跳设置页补全信息
- 没有模板、最近语言、封面预设、快速开始引导

建议：
- 提供“快速新建”和“完整新建”两条路径
- 默认模板可包含标题页、目录页、示例章节
- 对首次使用者增加空状态引导

### 2.2 书架信息密度低，缺少可管理性

状态：❌

位置：
- `src/pages/Home.vue`

问题：
- 书架只展示封面和极少操作
- 缺少搜索、排序、筛选、最近编辑时间、章节数等信息
- 大量书籍时会迅速失控

建议：
- 增加搜索栏、排序方式、列表/卡片视图切换
- 卡片上展示更新时间、作者、章节数
- 支持最近打开/收藏/继续编辑入口

### 2.3 编辑器工具栏功能堆叠，但信息架构不清晰

状态：❌

位置：
- `src/components/editor/EditorToolbar.vue`

问题：
- 工具过多，分组逻辑混杂
- 二级弹出项和一级按钮的优先级不稳定
- 一些功能依赖插入原始 HTML，实际是“技术实现”，不是“用户心智”

建议：
- 按“结构”“文本”“插入”“视图”“导出”重组
- 减少低频按钮常驻
- 把颜色高亮等高级功能折叠进“格式”面板

### 2.4 移动端体验只是“可用”，不是“为移动设计”

状态：❌

位置：
- `src/pages/Editor.vue`

问题：
- 当前移动端只是编辑/预览二选一切换
- 章节导航、插入、搜索、保存反馈、全屏状态都偏桌面思路

建议：
- 为移动端单独设计底部工具条和章节抽屉
- 提供“最近插入”“Markdown 片段”“输入辅助”
- 增强触屏拖拽和点击热区

### 2.5 预览主题和全局主题割裂

状态：❌

位置：
- `src/components/preview/MarkdownPreview.vue`
- `src/stores/theme.ts`

问题：
- 应用主题和预览主题各自独立、无持久化、无统一设定
- 用户很难形成“我当前的阅读/编辑环境是什么”的稳定认知

建议：
- 统一成“应用主题”和“预览纸张主题”两个明确概念
- 都做持久化，并在 UI 中给出可见反馈

## 3. 缺失功能

### 3.1 缺少导入能力

状态：❌

建议补充：
- Markdown 文件导入
- 文件夹批量导入章节
- 现有 EPUB / HTML / TXT 导入

原因：
- 当前只有“从零写作”和“导出”
- 真实用户通常先从已有稿件迁移

### 3.2 缺少恢复与历史能力

状态：❌

建议补充：
- 章节级历史版本
- 崩溃后恢复未完成编辑
- 手动快照 / 恢复点

原因：
- 自动保存只能降低损失，不能替代版本恢复

### 3.3 缺少目录与元信息增强

状态：❌

建议补充：
- 自定义目录样式
- 章节前言/后记/封面页控制
- 书籍元数据扩展：出版社、标签、版权页、简介 HTML

### 3.4 缺少资产管理

状态：❌

建议补充：
- 图片资源面板
- 已引用/未引用资源检查
- 图片复用、替换、压缩质量选项

### 3.5 缺少可用性保护

状态：❌

建议补充：
- 删除书籍/章节的撤销
- 导出前校验：空标题、重复标题、无章节、图片过大
- 离开编辑页时未保存状态确认

## 4. 可扩展性与代码质量问题

### 4.1 页面组件承担过多职责

状态：❌

问题：
- `Editor.vue` 同时处理路由、生命周期、保存、OCR、导出、同步滚动、全屏、章节 UI
- `Settings.vue` 同时处理表单、自动保存、路由退出、图片处理

计划：
- 拆出页面级 composable
- 页面保留编排，业务逻辑下沉

### 4.2 状态源过多，数据流不够清晰

状态：❌

问题：
- Pinia store、composable、本地 `ref`、`localStorage` 混用
- 偏好状态没有统一入口

计划：
- 归类为：业务数据、编辑会话状态、UI 偏好、平台状态
- 每类定义唯一读写入口

### 4.3 编辑器对外暴露接口过宽

状态：❌

位置：
- `src/components/editor/CodeMirrorEditor.vue`
- `src/components/editor/EditorToolbar.vue`

问题：
- 通过 `defineExpose()` 暴露大量命令式方法
- Toolbar 直接依赖这些方法的细节

影响：
- 替换编辑器实现会非常痛苦
- 工具栏与编辑器耦合过深

计划：
- 引入更窄的 editor command adapter
- Toolbar 只依赖稳定命令接口

### 4.4 数据层缺少事务化抽象

状态：❌

位置：
- `src/composables/useBook.ts`
- `src/composables/useChapter.ts`

问题：
- 跨表操作直接散落在 composable 内
- 删除书籍、删除章节、重排章节等操作缺少统一事务和约束校验

计划：
- 为书籍与章节建立 repository/service 层
- 复杂写操作统一封装

### 4.5 自动化测试覆盖面偏窄

状态：❌

已确认：
- 现有测试主要覆盖工具函数和局部 composable
- 页面、导出链路、Tauri 分支、Markdown 安全、拖拽树结构几乎没有覆盖
- 当前 `pnpm test -- --run` 在本环境下因 `vite/vitest` 启动时 `spawn EPERM` 失败，测试链路本身也不稳

计划：
- 修正测试命令和运行环境问题
- 增加：
  - Markdown 渲染安全测试
  - 章节树重排/非法移动测试
  - 导出成功/取消/失败测试
  - Settings 自动保存与离开页面测试
  - 关键页面组件测试

## 6. 推荐技能（skills）

### 适合当前项目的现有 skills

- `openai-docs`
  - 适用场景：如果后续要接入 AI 写作、润色、摘要、封面生成、章节扩写、智能校对
  - 价值：能用官方资料快速确定最新模型、接口和接入方式，避免过时方案

- `imagegen`
  - 适用场景：自动生成封面、章节插图、空状态插画、主题化书籍封面模板
  - 价值：这个项目天然适合做“快速出封面”的能力，能明显提升成品感

### 我建议你后续新增的自定义 skills

- `tauri-vue-desktop-audit`
  - 用途：专门审查 Tauri + Vue 桌面应用的菜单、窗口状态、快捷键、文件系统、权限和平台差异

- `epub-quality-check`
  - 用途：检查 EPUB 元数据、目录结构、封面、资源引用、样式兼容性、章节层级和阅读器兼容问题

- `editor-ux-review`
  - 用途：专门评估 Markdown 编辑器的快捷键、工具栏、滚动同步、移动端行为、输入体验和错误恢复
